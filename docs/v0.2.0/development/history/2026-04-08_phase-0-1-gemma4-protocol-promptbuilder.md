## Key Decisions

1. **Phase 0 before Phase 1**: The user chose to implement Phase 0 first rather than using a `ToolFormatStrategy` abstraction to decouple from Phase 0. This follows the plan's strict dependency order and avoids format mismatches.

2. **ToolCatalog as separate static data**: Tool metadata lives in `ToolCatalog.ts` as a static `TOOL_CATALOG` constant, separate from `ToolRegistry` (which handles runtime handler execution). This separation allows PromptBuilder to depend on metadata without requiring handler instances.

3. **ToolCallParser as thin re-export**: Rather than duplicating the Gemma 4 parsing logic, `ToolCallParser.ts` was converted to a pure re-export module from `Gemma4ToolFormat.ts`. This preserves the existing import paths used by AgentLoop and StreamingPipeline.

4. **ConversationManager accepts string, not PromptBuilder**: The constructor takes a plain `systemPrompt: string` rather than a `PromptBuilder` instance. This keeps ConversationManager as a pure conversation state manager. GemmaCodePanel owns the PromptBuilder and calls `rebuildSystemPrompt()` when context changes.

5. **Plan mode uses `rebuildSystemPrompt()` instead of `addSystemMessage()`**: Previously, toggling plan mode appended a separate system message that accumulated over time. Now it rebuilds the single system prompt in-place, which is cleaner and prevents message accumulation.

6. **PlanMode initialization order**: Had to move `this._planMode = new PlanMode()` earlier in the GemmaCodePanel constructor because `_buildPromptContext()` accesses `this._planMode.active` during initial prompt building.

## Deviations from Plan

1. **`formatToolResult` signature change**: The plan specified `formatToolResult(name, result)` but the old code used `formatToolResult(id, result)`. Updated AgentLoop to pass `call.tool` instead of `call.id`, since Gemma 4's tool result format uses the tool name, not a call ID.

2. **Ollama tool definitions built in GemmaCodePanel**: The plan mentioned threading tools through to API calls. Rather than a complex plumbing change, the tool definitions are built once in GemmaCodePanel and passed to both AgentLoop and StreamingPipeline via constructor parameters.

3. **No `client.ts` code changes**: The plan listed `client.ts` as needing updates, but since `streamChat()` already serializes the full `OllamaChatRequest` via `JSON.stringify`, simply adding `tools` to the request type was sufficient.

## Known Issues / Unresolved

- Python backend `prompt.py` still wraps the formatted prompt in a single user message for Gemma models (`assemble_prompt`). This may not be optimal with Ollama's native Gemma 4 support, where messages can be passed natively. Deferring to Phase 6 (backend alignment).
- The `<|"|>` string delimiter parsing in `Gemma4ToolFormat.ts` handles simple cases but may need hardening for string values containing the delimiter sequence itself. Real-world Gemma 4 output should not produce this edge case since the delimiters are special tokens, not literal text.
